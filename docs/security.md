# 운영 보안 기준

Hermes는 명령 승인, 파일 쓰기 보호, 컨테이너 격리, 메시지 사용자 인증 등 여러 겹의 방어 구조를 갖습니다. 그러나 실제 안전성은 **연결한 채널, 도구 권한, 비밀값 관리, 운영자의 승인 습관**에 달려 있습니다.

## 가장 먼저 지킬 네 가지

1. **비밀값을 Git에 올리지 않습니다.** API 키·봇 토큰·OAuth 자격증명은 `.env`에 보관하고 공개 저장소·대화·스크린샷에 넣지 않습니다.
2. **메시지 채널의 사용자 범위를 제한합니다.** 봇을 연결했다고 모든 DM과 그룹이 자동으로 신뢰되는 것은 아닙니다. 허용 사용자, DM pairing, 관리자 정책을 설정합니다.
3. **위험 명령은 사람이 승인합니다.** 기본 `smart` 정책을 유지하고, 자동화 환경에서 무분별한 전역 승인을 피합니다.
4. **프로필을 보안 경계로 오해하지 않습니다.** 프로필은 상태를 나누지만 로컬 파일 접근을 격리하지 않습니다.

## 명령 승인 정책

`config.yaml`의 `approvals`에서 위험한 명령에 대한 정책을 설정합니다.

```yaml
approvals:
  mode: smart
  timeout: 60
  cron_mode: deny
```

- `smart`: 위험도를 평가해 자동 승인·거부 또는 사람 확인으로 나눕니다. 기본값입니다.
- `manual`: 위험 명령마다 사람의 승인을 받습니다.
- `off`: 모든 승인 확인을 끕니다. 신뢰할 수 있는 격리 환경이 아니면 권장하지 않습니다.
- `cron_mode: deny`: 무인 예약 작업에서 위험 명령이 감지되면 막습니다.

## YOLO 모드는 운영 기본값이 아니다

`--yolo`, `/yolo`, `HERMES_YOLO_MODE=1`은 현재 세션의 위험 명령 승인 절차를 우회합니다. 일부 회복 불가능한 명령은 여전히 차단되지만, 이것을 안전장치로 생각하면 안 됩니다.

테스트용 컨테이너처럼 범위가 명확하고 되돌릴 수 있는 환경에서만 잠시 사용하고, 운영 gateway나 공개 채널에서 상시 활성화하지 마세요.

## 샌드박스와 파일 보호

로컬 터미널 백엔드는 OS 사용자 권한으로 명령을 실행합니다. 더 강한 격리가 필요하면 Docker, Singularity, 원격 SSH 등 용도에 맞는 백엔드를 검토합니다.

또한 쓰기 가능한 경로를 제한하는 정책은 실수나 프롬프트 인젝션의 피해 범위를 줄이는 데 도움이 됩니다. 단, 경로 제한만으로 네트워크·자격증명·외부 서비스 권한까지 보호되지는 않습니다.

## MCP를 연결할 때

MCP 서버는 외부 도구를 Hermes에 추가합니다. 설치 전에는 다음을 확인하세요.

- 누가 만든 서버인지, 소스와 설치 명령이 무엇인지
- 어떤 자격증명을 읽는지
- 실제로 필요한 도구만 노출하는지
- 삭제·결제·배포처럼 변경을 만드는 도구가 기본 선택돼 있지 않은지

MCP 카탈로그는 검토된 항목을 제공하지만, 설치되는 코드와 권한을 읽어 보는 책임은 운영자에게 있습니다.

## 사고를 줄이는 운영 습관

- 운영·테스트 프로필을 분리합니다.
- 외부 쓰기는 전용 채널과 명확한 승인 흐름을 둡니다.
- 정기적으로 `hermes config check`와 gateway 로그를 확인합니다.
- 토큰이 노출됐다고 의심되면 즉시 해당 제공자에서 폐기·재발급하고 `.env`를 교체합니다.
- 자동화는 읽기 전용에서 시작해, 필요한 경우에만 쓰기 권한을 추가합니다.
