# 운영 시작 전 체크리스트

입문 단계를 지나면 Hermes는 단순한 채팅 도구가 아니라 **모델·도구·파일·메시지 채널을 연결하는 실행 환경**이 됩니다. 운영의 목표는 기능을 많이 켜는 것이 아니라, 필요한 권한만 안전하게 제공하고 문제가 생겼을 때 원인을 찾을 수 있게 만드는 것입니다.

## 운영 순서

1. **기본 환경 점검** — 설치 뒤 `hermes status`, `hermes doctor`, `hermes config check`로 상태를 확인합니다.
2. **모델과 비용 정책 결정** — 기본 모델, 폴백 모델, 요청 시간 제한을 정합니다.
3. **도구 범위 축소** — 프로젝트마다 필요한 도구 세트만 켭니다.
4. **메시지 채널 접근 제어** — 공개 채널에 연결하기 전 허용 사용자와 DM 정책을 확인합니다.
5. **백그라운드 실행 점검** — gateway와 cron은 장시간 동작하므로 로그·알림·재시작 방식을 먼저 정합니다.
6. **업데이트 전 백업** — 운영 중인 설정과 상태를 보존하는 정책을 확인한 뒤 업데이트합니다.

## 운영자가 매일 확인할 것

- gateway가 살아 있는가?
- 예상하지 못한 cron 실행 또는 비용 증가가 없는가?
- 승인 대기 명령이 쌓이지 않았는가?
- API 키나 봇 토큰이 소스 저장소에 들어가지 않았는가?
- 특정 채널 또는 모델 제공자 오류가 반복되는가?

## 운영 원칙

### 최소 권한부터 시작

처음부터 모든 도구, 모든 경로, 모든 채널을 연결하지 마세요. 한 프로젝트와 한 채널에서 필요한 기능만 먼저 열고, 실제 사용 사례가 생길 때 확장하는 편이 안전합니다.

### 설정과 비밀값을 분리

모델 이름, 도구 정책 같은 일반 설정은 `config.yaml`에 둡니다. API 키·봇 토큰·비밀번호 같은 비밀값은 `.env`에만 보관하고 Git에 커밋하지 않습니다.

### 자동 작업은 독립된 작업으로 설계

cron 실행은 현재 대화 내용을 이어받지 않는 새 세션입니다. 따라서 "위 대화처럼"이 아니라 입력·출력·대상 채널·권한을 모두 적은 자기완결형 프롬프트가 필요합니다.
